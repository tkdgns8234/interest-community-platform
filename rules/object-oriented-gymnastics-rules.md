# 객체지향 생활 체조 9가지 원칙 - (ThoughtWorks Anthology)

> 이 원칙은 소프트웨어 장인정신을 기르기 위한 훈련법으로, 코드의 응집도를 높이고 결합도를 낮추는 것을 목표로 합니다.

> 이 원칙을 무조건적으로 지켜야 하는것은 아닙니다. 코드 복잡성과 유지보수성을 고려하여 적절히 적용하세요.

### 1. 한 메서드에 오직 한 단계의 들여쓰기만 허용하라
- **내용**: `if`, `for`, `while` 등의 제어문이 중첩되면 코드가 복잡해집니다.
- **방법**: 중첩이 생기면 해당 부분을 새로운 메서드로 추출하세요.

### 2. else 예약어를 쓰지 마라
- **내용**: `else`는 코드의 분기 경로를 늘려 가독성을 해칩니다.
- **방법**: **Early Return**(조건이 맞지 않으면 즉시 반환)을 사용하여 흐름을 단순화하세요.

### 3. 모든 원시값과 문자열을 포장하라
- **내용**: `int`, `String` 같은 기본 타입은 도메인의 의미를 담지 못합니다.
- **방법**: 의미를 담은 **값 객체(Value Object)**로 감싸서 스스로를 검증하게 하세요.

### 4. 한 줄에 점(.)을 하나만 찍어라
- **내용**: `a.getB().getC().do()`와 같이 객체 그래프를 깊게 탐색하는 것은 캡슐화를 해칩니다.
- **방법**: **디미터 법칙(Law of Demeter)**을 준수하여 인접한 객체에게만 메시지를 보내세요.

### 5. 줄여 쓰지 마라 (축약 금지)
- **내용**: `cnt`, `mgr` 등의 축약어는 의사소통 비용을 높입니다.
- **방법**: 이름이 길어지더라도 의도가 명확하게 드러나는 이름을 지으세요.

### 6. 모든 엔티티를 작게 유지하라
- **내용**: 클래스가 너무 크면 여러 책임을 가지게 됩니다.
- **방법**: 클래스는 50줄 이내, 패키지는 10개 파일 이내로 유지하는 것을 목표로 하세요.

### 7. 3개 이상의 인스턴스 변수를 가진 클래스를 쓰지 마라
- **내용**: 인스턴스 변수가 많을수록 클래스의 응집도가 낮아집니다.
- **방법**: 연관된 필드들을 묶어 새로운 객체로 분리하세요.

### 8. 일급 컬렉션을 사용하라
- **내용**: 컬렉션을 포함하는 클래스는 그 컬렉션 외에 다른 멤버 변수가 없어야 합니다.
- **방법**: 컬렉션을 전용 클래스로 감싸서 관련 로직을 한곳에서 관리하세요.

### 9. 게터(Getter)/세터(Setter)/프로퍼티를 쓰지 마라
- **내용**: 객체에서 데이터를 꺼내지 말고 객체에게 일을 시키세요.
- **방법**: **Tell, Don't Ask** 원칙을 따라 객체의 상태 변경 주도권을 객체 내부에 두세요.
